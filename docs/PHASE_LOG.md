# sciknow — Phase log

Release notes, newest first. Each entry documents a phase commit: what
changed, why, and where it landed. Mirrors what used to live inline
in the README but is now isolated so the README stays focused on
"how sciknow works today" rather than "how it got here".

Each entry's version number matches the Phase commit tag in
``git log --oneline`` (e.g. ``54.6.82`` → commit subject
``feat(retrieval): 54.6.82 — ...``).

> **Phase 54.6.83 (#8) — offline claim-atomization verifier.** Sentence-level NLI (what the quality bench's `faithfulness_score` and the autowrite verifier use) has a known blind-spot: *"X causes Y, and Greenland observations confirm it"* gets one entailment score, but the two claims are independent and the sources may support only one. New `sciknow/core/claim_atomize.py` atomizes sentences into atomic sub-claims (heuristic-first — regex on `;`, em-dashes, clear `, and` clause boundaries — LLM fallback only for long multi-conjunction sentences so the average cost stays near zero), NLI-scores each sub-claim against the source chunks, and reports per-sentence `mixed_truth` flag (≥1 supported AND ≥1 unsupported). New CLI `sciknow book verify-draft <draft_id>` surfaces the mixed-truth sentences with per-sub-claim ✓/✗ flags. Read-only; reads `drafts.sources`. Not wired into the quality bench (doing so would multiply NLI cost 2-3× for marginal signal). Reference: FActScore (Min 2023) — RESEARCH.md §526 keeps the offline slice open.

> **Phase 54.6.82 (#11 follow-up) — visuals Qdrant index + semantic search.** Closes the loop on 54.6.78 (equation paraphrase) + 54.6.72 (figure captions): both land prose in `visuals.ai_caption`, but nothing was indexing it for similarity search. New `sciknow db embed-visuals` embeds those captions into the per-project visuals Qdrant collection (created in Phase 21.b) via bge-m3 dense + sparse. New `retrieval/visuals_search.py` with `search_visuals(query, kind=...)` RRF-fuses the two signals using the modern `qdrant_client.query_points` API. New `GET /api/visuals/search?q=...&kind=equation&k=10` returns ranked hits joined to the full visuals row (asset_path + paper_title + year) so the GUI can render thumbnails alongside the AI caption. Smoke: embedded 3 equation paraphrases, queried *"outgoing longwave radiation slope"* — correctly ranked OLR equation (0.033) > dOLR/dT derivative (0.032) > MEI index (0.016). Full backfill needs `paraphrase-equations` + `caption-visuals` to have run first.

> **Phase 54.6.81 (#10 part 2) — paper-type weighted retrieval (default OFF).** Completes #10. `hybrid_search` multiplies each candidate's `rrf_score` by a paper-type-specific weight when `paper_type_weighting=true`: peer_reviewed / preprint / thesis / book_chapter = 1.0, editorial / policy = 0.7, unknown = 0.8, **opinion = 0.4** (hardest cut). Default OFF until the 54.6.80 classifier backfill completes on a meaningful fraction of the corpus — otherwise everything uses the 'unknown' fallback and rankings shift for a spurious reason. Users enable with `PAPER_TYPE_WEIGHTING=true` in `.env`; per-type weights overridable via `PAPER_TYPE_WEIGHTS`. SearchCandidate gets a new `paper_type` field populated by the apply-function.

> **Phase 54.6.80 (#10 part 1) — paper-type classifier + schema.** Corpus mixes peer-reviewed papers, preprints, theses, policy briefs, opinion pieces, editorials, and book chapters at equal retrieval weight. For factual queries an opinion piece is a liability. Migration 0027 adds `paper_type` + `paper_type_confidence` + `paper_type_model` columns to `paper_metadata`, with a partial index on `paper_type IS NOT NULL` so filter queries stay fast. New `sciknow/core/paper_type.py` classifier takes title + journal + publisher + year + DOI + abstract + first ~2000 chars of content and returns one of 8 categories with confidence 0.0–1.0. Handles alias fallbacks (`peer-reviewed` → `peer_reviewed`, `op-ed` → `opinion`, etc.) and degrades to None on garbage JSON so re-runs can retry. CLI `sciknow db classify-papers [-n N] [--force]` at ~2-3s/paper on LLM_FAST_MODEL. Live smoke on 3 papers: *"IPCC's Earth Energy Imbalance..."* → opinion (0.95), *"Atmospheric Effects"* → peer_reviewed (0.95), *"Higgs boson excesses..."* → peer_reviewed (0.95).

> **Phase 54.6.79 (#6) — plan-coverage dimension in the autowrite scoring loop.** Until now, autowrite judged drafts on prose dimensions (groundedness, citation accuracy, coherence, hedging, length) but never *"did the draft cover the section plan?"*. A draft can score 0.9 on every prose dimension while silently skipping half the plan bullets. New `sciknow/core/plan_coverage.py` atomizes the plan text (sentence-split, fallback to semicolon/pipe), entails each bullet against the draft via the already-loaded NLI cross-encoder (no new model load), returns a coverage fraction + per-bullet scores + missed-bullets list. Folded into the autowrite `scores` dict as a new `plan_coverage` dimension — the existing weakest-dimension + KEEP/DISCARD logic picks it up automatically. When coverage is the weakest dimension AND below `target_score`, the revision instruction is overridden with a hint naming the missed bullets explicitly ("The section plan calls for these points that the current draft doesn't cover: …"). Reference: CovScore (Zhang 2024).

> **Phase 54.6.78 (#11) — equation natural-language paraphrase for retrieval.** bge-m3 embeds raw LaTeX poorly because the tokenizer fragments `\frac` and similar commands into character-level pieces — the resulting embedding drifts from the equation's semantic meaning. A one-sentence prose paraphrase (*"The slope of outgoing longwave radiation with respect to global surface temperature, 2.93 ± 0.3 W/m²·K"*) embeds far better. New `sciknow/core/equation_paraphrase.py` + `sciknow db paraphrase-equations` CLI. Uses LLM_FAST_MODEL with a retrieval-targeted prompt (one sentence, 15-50 words, keep variable names + numeric values with units, translate LaTeX into prose). Skips trivial scalar identities (`a=b`). Stores in the existing `visuals.ai_caption` column (54.6.72 migration). Smoke on 3 equations: `$OLR(ΔT)=OLR₀+(dOLT/dT)·ΔT$` → *"The change in outgoing longwave radiation, OLR, as a function of temperature change, ΔT, is given by the initial OLR value, OLR₀, plus the product of the derivative dOLR/dT and ΔT."* ~2.2h for all 4,687 equations on a 3090.

> **Phase 54.6.77 (#16) — MCP server exposes the corpus to external agents.** New `sciknow mcp-serve` speaks Model Context Protocol over stdio; any MCP-aware agent (Claude Desktop, Claude Code, goose, etc.) can query the corpus directly. > **Phase 54.6.71 (#7) — citation marker → chunk alignment post-pass.** Directly targets the autowrite_writer citation_precision failure mode from the 2026-04-17 quality bench (11% for gemma3, 60% for qwen — 40-90% of emitted `[N]` markers didn't point at the right source). New `sciknow/core/citation_align.py` module: after the writer produces a draft, for every sentence with `[N]` citations, score the sentence's entailment against every retrieval source chunk via the already-loaded NLI cross-encoder (`cross-encoder/nli-deberta-v3-base`). If the claimed chunk N entails below 0.5 AND another chunk M entails above it by ≥0.15 margin, remap `[N]` → `[M]`. Multi-citation markers `[1, 5]` are handled per-number so a correct `[1]` stays when only `[5]` is wrong; post-remap dedup collapses `[1, 1]` to `[1]`. Wired inline into `_autowrite_section_body` between writer and scorer so the scorer grades the corrected prose, never the raw writer output. Also exposed as an on-demand `sciknow book align-citations <draft_id>` CLI for after-the-fact alignment. L1 uses a mocked NLI scorer to exercise real remap logic without the 440MB model load. Reference: LongCite (Zhang et al., 2024). Zero new LLM cost — reuses the quality-bench NLI model.

> **Phase 54.6.70 (#9) — co-citation / bibliographic-coupling retrieval boost (default OFF).** New `_apply_cocite_boost` in `hybrid_search`: within the retrieved set, count citation-graph edges (either direction) between each candidate's `document_id` and the top-20 anchor set, then apply `rrf_score × (1 + 0.1 × log2(1 + n_edges))`. Topic-local signal, distinct from the global-popularity `_apply_citation_boost` already shipped in Phase 32.8. Applied after `_apply_useful_boost` so citation × useful × cocite compose multiplicatively. **A/B on the 54.6.69 retrieval bench regressed MRR 0.514 → 0.496 on the 30-query smoke set** — with ~4.5% in-corpus citation resolution the signal is sparse and biased toward popular papers rather than true-topic neighbors, so `cocite_boost_factor` defaults to 0.0. Implementation stays; users opt in via `.env` when in-corpus density improves. Lesson: the retrieval bench from 54.6.69 paid off within hours — it caught a regression before it shipped. References: Kessler 1963 (bib-coupling), Small 1973 (co-citation). NOT GraphRAG-global (rejected in RESEARCH.md §526) — different axis, different scope.

> **Phase 54.6.69 (#3) — retrieval-quality benchmark harness (MRR / Recall / NDCG).** Bench layer previously measured only retrieval *speed* + *rerank impact* — no accuracy metric. New `sciknow/testing/retrieval_eval.py`: `generate_probe_set()` samples N chunks (≥400 chars) from the corpus with a deterministic seed, asks `LLM_FAST_MODEL` to write one specific question per chunk ("answerable ONLY from this passage, 8-25 words, no answer reveal"), persists `(question, source_chunk_id, source_qdrant_point_id)` to `<project>/data/bench/retrieval_queries.jsonl`. New `b_retrieval_recall` bench function loads the file, runs hybrid search for each query, and computes **MRR@10, Recall@1, Recall@10, NDCG@10, not_found_pct, latency_p50/mean** — registered in `_LIVE` via a lazy wrapper so `fast` runs don't pay the import. New CLI `sciknow bench-retrieval-gen -n 200` for one-time probe-set generation (`~2-5 min LLM_FAST_MODEL time`). `_find_source_rank` matches candidate's `chunk_id` against the stored `qdrant_point_id` (SearchCandidate.chunk_id IS the qdrant UUID, not PG chunks.id — easy trap). Baseline on the Global Cooling corpus (30-query smoke, 24.5k chunks): **MRR@10 0.514, Recall@1 0.33, Recall@10 0.77, NDCG@10 0.58, not_found 17%**. Regression baseline for every subsequent retrieval tweak. Already paid off catching the 54.6.70 regression.

> **Phase 54.6.65 — data-weighted section counts in `book outline`.** The outline's section-count-per-chapter used to collapse to whatever number the LLM picked first (5 on the 2026-04-18 run for every chapter of Global Cooling, then 3 on the re-run after the variance-nudge — *uniform, just at a different level*). The variance was cosmetic even when the prompt asked for 3–6, because (a) nothing told the LLM that chapter scope varies with actual corpus evidence, and (b) the CLI tree-search scorer was monotonic in total-section-count, rewarding "max sections × all chapters" over varied candidates. Now: (1) the outline prompt opens to 3–8 with explicit guidance to vary with scope and evidence depth (not uniform); (2) the CLI candidate scorer adds a section-count standard-deviation term (×2.0 weight) so varied candidates beat uniform-max; (3) a new shared helper `core/book_ops.py::resize_sections_by_density(chapters)` runs one hybrid retrieval per chapter on `topic_query`, counts distinct papers in the top-100, buckets to a target count (≤3→2, 4-8→3, 9-20→4, 21-40→5, 41-70→6, 71+→7), then **trims over-specified chapters and grows under-specified ones by ≥2** (fast-model call for "propose N additional complementary sections"). Both the CLI (`book outline`) and the web reader's outline handler route through the helper. Each chapter now displays a density note: `Density: 53 papers → target 6 (grown (+2) from 4 → 6)`. Live re-run on the 676-paper Global Cooling corpus went from 9 × 3 = 27 uniform sections to a varied 4/4/5/6 mix — chapters with ≥41 papers got the full 6, sparser chapters stayed at 4.

> **Phase 54.6.64 — replace `:name::type` with `CAST(:name AS type)` in five INSERTs.** User reported `sciknow book outline "Global Cooling"` crashed with *`ProgrammingError: syntax error at or near ":"`* at `:secs::jsonb`. Root cause: SQLAlchemy's `text()` uses `:name` for named parameters, Postgres uses `::type` for casts, and the two collide when you write `:name::type` — SQLAlchemy translates the leading `:name` but leaves the trailing half of `::type` as literal SQL, producing `%(name)s::jsonb` in some contexts and malformed `:name::type` in others (which is exactly what we saw). The codebase already had fix notes at `book_ops.py:1629` and `web/app.py:717` explicitly telling future contributors to use `CAST(:x AS jsonb)`, but 5 sites slipped through: `cli/book.py:327` (draft insert in `book write`), `cli/book.py:806` (chapter insert in `book outline --save` — the reported one), `cli/ask.py:375` (ask-write drafts), `web/app.py:615` (`draft_comments` insert), `web/app.py:1928` (`draft_snapshots` insert). All five flipped to `CAST(:x AS jsonb)` / `CAST(:did AS uuid)`; remaining `:name::type` hits are the existing comment warnings themselves. Verified end-to-end by re-running `sciknow book outline "Global Cooling"` — 8 chapters saved cleanly.

> **Phase 54.6.60 — RAPTOR already-built is a benign skip, not an error.** The RAPTOR build genuinely isn't resumable (each level is a global UMAP → GMM → BIC fit on the full level-(N−1) node set, and "resume" would mean either a full rebuild or stale nearest-neighbor assignment — raptor.py's own docstring calls this out: *"The build is one-time-batch; existing summary nodes stay useful but slightly stale when new papers are ingested. Re-run with --rebuild periodically."*). The error was the wrong UX though — re-running `sciknow refresh` on a corpus with an already-built tree landed on `exit 1` with "RAPTOR summary nodes already exist. Pass --rebuild", which broke refresh's idempotent-resume contract and printed a red `⚠ 5. RAPTOR tree build failed` warning even when nothing was wrong. Fixed: raptor.py now yields an `already_built` event with per-level node counts instead of an `error` event; the CLI handles it as a success, prints `↷ RAPTOR tree already built (62 summary nodes: L1=50, L2=10, L3=2)` with the one-time-batch disclaimer, and exits 0. Refresh re-runs now report RAPTOR as ✓ when the tree exists. To absorb newly-ingested papers into summaries, still pass `--rebuild` explicitly (unchanged).

> **Phase 54.6.59 — `AUTOWRITE_SCORER_MODEL` role override wired (gemopus4 specialist).** Added `autowrite_scorer_model` setting on `Settings`, routed through `_autowrite_section_body` so the autowrite score + rescore LLM calls run on `gemopus4:26b-a4b-q4_K_M` while everything else (writer, revise, verify, CoVe, plan, step-back) stays on the unified qwen `LLM_MODEL`. The 2026-04-17-full bench showed gemopus4 gives a **ranking gap of 0.60** on the good/bad draft discrimination task vs qwen's 0.27 — both rank correctly but gemopus4 is decisive (good=0.75, bad=0.15) where qwen hedges (good=0.87, bad=0.60), at the same 6-7s per scorer call. gemopus4 is broken on the other 9 quality tasks (hung for 2,656s on wiki_consensus returning empty, 0 words on 6/10 tasks), so it's wired here ONLY.
