# sciknow — Phase log

Release notes, newest first. Each entry documents a phase commit: what
changed, why, and where it landed. Mirrors what used to live inline
in the README but is now isolated so the README stays focused on
"how sciknow works today" rather than "how it got here".

Each entry's version number matches the Phase commit tag in
``git log --oneline`` (e.g. ``54.6.82`` → commit subject
``feat(retrieval): 54.6.82 — ...``).

> **Phase 54.6.118 — Tier 4 #3: corpus-drift detection.** New module `sciknow/retrieval/corpus_drift.py` with `record_drift(project_root, tag, reason, also_size)` — takes the current bge-m3 abstract centroid, compares to the previous snapshot (1-cosine), appends a line to `<project data>/expand/drift.log`, overwrites `drift.latest.json` for the next diff. `read_recent(n)` parses the tail for display. Hooked into `db expand` so every expand round writes a drift record with the relevance_query as the reason; the CLI colour-bands the output (`dim < 0.01 < yellow < 0.05 < red`) so the user can see when an expansion pulled the corpus noticeably toward a new subtopic. New CLI: `sciknow db drift [-n 10] [--snapshot --tag T --reason R]` for manual snapshots and tail viewing. Live-tested end-to-end: two consecutive snapshots on an unchanged corpus gave delta=0.0000 as expected.

> **Phase 54.6.114 — Tier 2 #2: question-driven agentic expansion.** A new capability class: `sciknow db expand --question "..."` lets you hand sciknow a research question instead of a seed corpus. The LLM decomposes it into 3-6 narrow sub-topics; for each, hybrid_search measures coverage; sub-topics with < N papers are "gaps" and get a targeted Phase-49 RRF expansion (reusing everything from 54.6.70-113 — co-cite, bib-coupling, PageRank, isInfluential, MMR diversity, citation-context embeddings, venue block/allow); after ingest, coverage re-measured; loop until every sub-topic is covered OR coverage stalls (triggers an LLM re-plan) OR max_rounds. New module `sciknow/ingestion/agentic_expand.py`: `decompose_question(question, coverage?)` (refine mode when coverage supplied), `check_coverage(subtopics, doc_threshold=3)`, generator `run_agentic_expansion(...)` that yields structured events (`decomp / coverage / round_start / round_done / stopped / error`). Decomposition output validated live against Ollama on the test question "What is the current best estimate of ECS?" → 5 crisp sub-topics ("equilibrium climate sensitivity paleoclimate estimates", "model intercomparison", "observational constraints", "uncertainty quantification", "CMIP6 model results"). Per-gap expansion runs as a subprocess calling `db expand --relevance-query <gap>` so the Qdrant/DB/asyncio state stays clean. New CLI flags: `--question` (mutually exclusive with `--relevance-query`), `--question-rounds` (default 3), `--question-budget` (per-sub-topic, default 10), `--question-threshold` (coverage floor, default 3). GUI: new **Agentic (question-driven)** tab in the Corpus modal with a textarea for the question, numeric inputs for rounds/budget/threshold, and a dry-run checkbox. Streams plan + coverage + per-sub-topic progress into the existing corpus log via the 54.6.111 `runCorpusCliAction` helper.

> **Phase 54.6.113 — Tier 2 #1: citation-context embeddings in RRF (biggest research upgrade in the backlog).** Cohan et al. 2024 (SciRepEval) finding: a paper's abstract is often "marketing prose" but the short sentences that OTHER papers wrote when citing it — Semantic Scholar's `contexts` field — are how the community actually uses it, and they're discriminative for relevance in a way abstracts aren't. New module `sciknow/retrieval/citation_context.py`: `extract_contexts(s2_citations, max_contexts=20)` flattens + dedupes, prioritising contexts from influential citations; `embed_contexts(contexts)` concatenates with `[SEP]` and runs bge-m3 (one encode, not N averaged, to stay cheap); `context_cosine(vec, centroid)` safe against None inputs. Wired into `_run_rrf_ranker` step 8 alongside the existing S2 fetch (zero extra API calls — we were already fetching contexts but discarding them). Added `citation_context_cosine` + `citation_context_n` fields to `CandidateFeatures`; added as the 8th RRF signal (complementary to bge_m3_cosine — new uncited papers have no contexts, old papers with bad abstracts have no abstract signal); surfaced in the shortlist TSV. Small-sample unit smoke confirms the influential-first ordering + dedup logic. Full corpus benchmark pending — next `sciknow db expand` run on the global-cooling corpus will produce the first real numbers.

> **Phase 54.6.108 — ROOT CAUSE: the 600-char client-side truncation broke long equations. Removed.** User confirmed /debug/equations renders perfectly but the Visuals modal still fails half the equations. That pinpointed the failure to the modal's render path, not to KaTeX or the data. Compared the two paths: the diagnostic renders FULL equation bodies, the modal truncates to 600 chars and appends `…`. Verified against live API on the first page of equations: **3 of 12 equations (25%) are > 600 chars** — all `\begin{array}` multi-row blocks, multi-line sums, piecewise-defined functions. On recent-first ordering (which surfaces the complex newer papers first), the failure rate approaches half — matches the user's report. Fix: deleted the `body.length > 600 ? body.substring(0, 600) + '…' : body` guard. The server API already caps `content` at 2000 chars server-side; no client-side truncation needed. KaTeX display mode with `overflow-x:auto` handles wide formulas via horizontal scroll in the card, so removing the cap doesn't break layout. Five phases to find what should've been caught in 54.6.101 — lesson: when the diagnostic shows 100% render success, look for a cap the diagnostic skipped.

> **Phase 54.6.105 — direct `katex.render()` on each equation, no more auto-render roulette.** User: "lot of formulas still not being displayed correctly, please do a whole debug of this because its the fourth time". Actually debugged this time instead of guessing: ran 600 real equations through the vendored KaTeX offline — **596/600 render cleanly** at the LaTeX level (only 3 `\displaylimits` and 1 malformed-array failures, all within `throwOnError:false` recoverable territory). So the LaTeX itself is fine; the four prior fixes (54.6.101-103) were chasing auto-render fragility. Root cause: KaTeX's auto-render walks the live DOM matching `$$...$$` delimiters against text-node boundaries — any adjacent HTML structure, escaped entity, or innerHTML-parsing quirk that changed the text-node layout could drop the match and leave raw `$$…$$` visible even though KaTeX itself was loaded and the LaTeX was valid. Fix: dropped auto-render entirely. Equation cards now render into a `<div class="eq-target" data-latex="…">` and a post-innerHTML pass calls `window.katex.render(el.dataset.latex, el, {displayMode:true, throwOnError:false, output:'html'})` directly on each one. Attribute values are exempt from HTML parsing so `<` and `>` in the LaTeX survive intact; no delimiter matching needed because we know exactly which elements carry equations. Backoff retry every 200ms (up to 10×) for the case where `window.katex` isn't global yet. Applied to both the Explore > Visuals modal and the wiki Visuals tab. Also added a `#visuals[=kind]` URL-hash hook that auto-opens the modal — useful for future share-links and for headless testing.

> **Phase 54.6.103 — HTML-escape equation LaTeX so embedded `<` doesn't break the renderer.** User reported after 54.6.102 that *some* equations still rendered raw, specifically "the ones who doesnt start with a `$$` string". Diagnosis: 100% of equations in the DB start with `$$` (verified), so the user was observing the rendered result — cards with the visible `$$` wrapper are the ones KaTeX failed on. Root cause: LaTeX math frequently contains `<` (64 rows with examples like `- \infty < x < \infty`, `z _ { i } < \zeta _ { 1 }`); when that landed inside an `innerHTML` assignment the browser's HTML parser treated `<x` as an unknown-tag start and mangled everything around it — the `$$` delimiters were left stranded outside the broken DOM fragment and KaTeX never matched them. Fix: run `_escHtml()` over the LaTeX body before injecting between `$$…$$` in both the Explore > Visuals modal and the wiki Visuals tab. The browser decodes `&lt;` back to `<` when creating the text node, which is exactly what KaTeX's auto-render inspects. Same fix applied to the wiki tab's equation path (the 2023-era code had the same bug). Also bumped the wiki tab's equation card to the same visual polish as Explore (white background, 17px display math, `.vis-eq` class).

> **Phase 54.6.102 — equations finally render; tables get grid lines + zebra rows.** After 54.6.101 user still saw raw LaTeX. Root cause: MinerU stores equation `content` **already wrapped in `$$...$$`** (verified against the `visuals` table — every equation row starts and ends with `$$\n...\n$$`). 54.6.101 then wrapped it AGAIN, producing `$$$$<latex>$$$$` which KaTeX auto-render can't match. Fix: strip any existing LaTeX delimiters (`$$`, `$`, `\\[ \\]`, `\\( \\)`) from `v.content` before re-wrapping with exactly one `$$…$$` pair. Added a short retry (doRender at 0/250ms/800ms) for the case where the `defer`-loaded KaTeX auto-render script is still pending when the modal opens. Tables: added scoped `.vis-table-wrap` CSS — `border-collapse`, 1px grey borders, zebra rows, bold header background, caption-side top. Scoped so it doesn't leak into the heatmap or other tables on the page. Also added `.vis-eq .katex` font-scaling so the 17px display math inherits KaTeX's proportional sizing correctly.

> **Phase 54.6.64 — replace `:name::type` with `CAST(:name AS type)` in five INSERTs.** User reported `sciknow book outline "Global Cooling"` crashed with *`ProgrammingError: syntax error at or near ":"`* at `:secs::jsonb`.
