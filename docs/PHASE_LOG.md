# sciknow — Phase log

Release notes, newest first. Each entry documents a phase commit: what
changed, why, and where it landed. Mirrors what used to live inline
in the README but is now isolated so the README stays focused on
"how sciknow works today" rather than "how it got here".

Each entry's version number matches the Phase commit tag in
``git log --oneline`` (e.g. ``54.6.82`` → commit subject
``feat(retrieval): 54.6.82 — ...``).

> **Phase 55.V1 — VRAM discipline + cross-family scorer routing fix.** Two-part drop addressing user feedback that gemma-4-31B was loading on top of the embedder + reranker (the writer was correctly evicted by the Phase 55.S1 conflict map, but the small retrieval roles weren't — pinning ~1.8 GB at the worst possible time). The user's directive: "always when using either the writer or the scorer unload any other model from the vram." **Implementation.** `_VRAM_CONFLICTS` in `sciknow/infer/server.py` extended bidirectionally — writer/scorer/vlm now each conflict with embedder + reranker, and vice-versa. Embedder ↔ reranker still co-reside (intentional retrieve-phase pair). New `activate_phase("retrieve" | "generate" | "score" | "vlm")` API brings up the required role(s) and evicts everything else with the existing 1.5 s CUDA-context grace; `hot_phase` is the context-manager flavour. New `_swap_to_phase` bridge in `core/book_ops.py` routes to activate_phase in v2 and falls through to the legacy `_release_gpu_models` + `release_llm` dance in v1 (rollback path). The autowrite engine (`core/autowrite.py`) gained ~10 explicit phase swaps at every retrieve↔generate↔score boundary: initial retrieve, planning, lessons fetch, write, score, verify, CoVe, auto-expand, adaptive_retrieval, revise, rescore. The scorer's `ctx_size` was restored 8192 → 16384 (the KV-cache hack from earlier in the same day is no longer needed once the embedder/reranker are evicted). New `VRAM_CO_RESIDENCE_OK=true` setting short-circuits the eviction policy on hosts where the math works (DGX Spark, A100 80 GB). **Scorer-routing regression caught during 55.V1 verification.** `_client_for(role)` in `sciknow/infer/client.py` was missing the `"scorer"` URL-map entry, so any call with `role="scorer"` raised `KeyError "scorer"` inside `chat_stream`, which got swallowed by the autowrite scoring stage's broad `except` and silently fell back to `overall=0.5`. The 1-iter end-to-end test went from `score 0.50` (silent failure) → `score 0.89` (correct) once the URL map was patched. **Monitor alert generalisation.** `_build_alerts` in `core/monitor.py` now demotes ANY infer_* role being down to `info` severity (not warn/error) whenever a peer big-model role is up — the auto-swap is expected mid-cycle, not a service outage. Generalised from the writer↔scorer pair to all `(writer|scorer|vlm) × (embedder|reranker)` combinations. Three new L1 contract tests pin the conflict-map cartesian, the per-phase swap-call counts (≥4 score, ≥3 generate, ≥5 retrieve), and the scorer-role routing surface (`role="scorer"` count ≥ 4, `_client_for` URL map covers all five roles). **Verified end-to-end on the 3090.** 1-iter run: 1836 words, score 0.89, gemma decode at 30 t/s during scoring (full speed, no partial-load). 2-iter run: 1803 words, final 0.87. VRAM observed: writer alone @ 18.9 GB, gemma alone @ 22.4 GB, embedder + reranker alone @ 1 GB during retrieval — never co-resident with a big model. Future work tracked in `docs/ROADMAP.md` §8: Phase 55.V2 (router-mode migration via `llama-server --models-dir`) and Phase 55.V3 (propagate `_swap_to_phase` to wiki/extract-kg/argue/gaps engines).

> **Phase 54.6.313h — optional `--llm-fallback` layer (opt-in last resort).** New `llm_recover_title(pdf)` in `sciknow/ingestion/enrich_sources.py` reads the first 3 pages of the PDF, ships them to `settings.llm_fast_model` with the narrow prompt "return ONLY the paper's title or the literal string NO_TITLE", sanity-checks the output (10–300 chars, strips common wrappers, rejects "unknown"/"n/a"), and feeds the result into the existing title-search cascade. Wired as a last-resort layer in `_lookup`, gated on the new `--llm-fallback` CLI flag (default off). Source tag `llm_recovered_title` is distinct from the 54.6.313b `recovered_title` so the per-source breakdown can tell the two apart. On the global-cooling residual 154-paper set: **0 new matches** via LLM fallback — the residuals are genuinely not in any DOI index (Spanish INGEMMET government volcanology reports, climate-skeptic blog posts, AIAA conference proceedings, pre-DOI-era works). Feature still ships because it will help other corpora with mainstream papers that have garbled DB titles. Slow (~5–15 s per row via Ollama `keep_alive=-1`), so the recommended workflow is `sciknow corpus enrich` for the zero-cost pass then `sciknow corpus enrich --llm-fallback` for the long tail.

> **Phase 54.6.313g — filename-DOI extraction layer (+6 matches).** The sciknow downloader persists DOI-fetched PDFs as `<doi>.pdf` with slashes replaced by underscores (`10.1017/s1743921312005066` → `10.1017_s1743921312005066.pdf`). When the ingest metadata pipeline later lost the DOI linkage (arxiv-preprint pair split, metadata cascade fell through to LLM layer without the DOI signal), the filename still carries the authoritative identifier. New `extract_filename_doi(pdf)` parses via `10\.\d{4,5}[_\-](.+?)(?:\.pdf)?` and validates via Crossref `/works/{doi}` before accepting. Placed FIRST in the PDF-read layer order (before XMP + fulltext-regex) — cheapest signal + highest precision when present. meta.source tag is `filename_doi`. L1 regression test extended to pin the new adapter + tag. Dry-run on 161 residuals (after 54.6.313f): +6 matches (`crossref+filename_doi 6`). Cumulative 67/217 = 33.5× baseline.

> **Phase 54.6.313f — garbage first_author names poisoning title-search.** Bug discovered while iterating on residuals: `search_crossref_by_title` was passing the DB's `first_author` value as Crossref's `query.author` filter whenever the string was non-empty. That narrows Crossref's search to papers whose author list contains that string — perfect for real names, catastrophic when the DB's author was a PDF Info-dict artefact like "Usuario" (Spanish "user"), "Propietario", "Benutzer", "ASUS" / "Dell" / "HP" (OEM defaults), "hy", "Print KDP", etc. Two fixes in `metadata.py`: (1) expanded `_GARBAGE_AUTHOR_RE` to cover Spanish/Portuguese/German/French/Italian equivalents of "user"/"owner" and common OEM strings; (2) `search_crossref_by_title` and `search_openalex_by_title` now skip passing `first_author` as a search filter when `_is_garbage_author` is True OR the name is <3 chars. The author is still used as an *accept* signal (dual-signal validation), just not a *filter*. Verified: "Sunspot characteristics at the onset of the Maunder Minimum based on the observations of Hevelius" (DB author="Usuario") now correctly matches Crossref DOI `10.3847/1538-4357/ab4ade`. Previously the Usuario filter was ranking "Deep-Sea Sounding and Deep-Sea Thermometers" at the top (a paper whose author list happens to contain a token Crossref hashed to "usuario"). Measured on the 172 residual no-DOI set after 54.6.313e: +12 matches dry-run, +11 live-persisted (10 `crossref`, 2 `datacite`, also unexpectedly converted all 5 `arxiv_id_in_title` stubs from bare-arxiv-id to proper DOIs — 3 via Crossref journal-version, 2 via DataCite's 10.48550/arxiv prefix).

> **Phase 54.6.313e — arxiv-id-in-title fast path + `_ARXIV_OLD` regex fix.** Follow-up that caught 5 more matches on the residual no-DOI set, pushing the total from 45 → 50 (=25× the pre-change baseline). Bug: the `_ARXIV_OLD` pattern `\b([a-z\-]+/\d{7})\b` in `sciknow/utils/doi.py` failed on titles like `arXiv:astro-ph/0207637v1  29 Jul 2002` because `\b` doesn't match between `7` and `v`. > **Phase 54.6.313 — `db enrich` top-ROI improvements from ENRICH_RESEARCH.md: PDF XMP / fulltext regex / title recovery / S2 /match / Europe PMC / arXiv title / DataCite.** Implementation phase for the research memo shipped in 54.6.312.5. New module `sciknow/ingestion/enrich_sources.py` with six source-adapter functions; the enrich CLI cascade at `sciknow/cli/db.py::enrich._lookup` now tries them in order (first-hit-wins) before giving up. **(1) `extract_xmp_doi(pdf)`** — PyMuPDF's `doc.xref_xml_metadata()` + `doc.xref_stream(xref)` reads the raw XMP packet; four regex patterns extract `prism:doi`, `dc:identifier`, `prism:url`, and a DOI-shaped fallback (catches publishers that use non-standard tags). **(2) `extract_fulltext_doi(pdf, max_pages=3, validate=True)`** — regex-scans the first-3-page extracted text for `10.\d{4,9}/\S+`, collects up to 5 candidates (first hit is sometimes a *cited* paper, not the current one), and validates each via Crossref `/works/{doi}` with `validate_doi_resolves` to reject OCR-mangled strings. **(3) `recover_title_from_pdf(pdf)`** — fallback for garbage-titled rows: PyMuPDF dict-layout walk finds the largest-font line in the top 40 % of page 1 and returns it as a search title so Crossref/OpenAlex/S2/arXiv/DataCite can have a second try. **(4) `search_semantic_scholar_match(title, first_author, year)`** — the purpose-built `/graph/v1/paper/search/match` endpoint (different from `/search`). Process-wide `_S2_LOCK` + `_s2_wait_for_token(1.1s)` token bucket ensures parallel workers share politely across the unauth 1 RPS pool; 429 responses trigger exp. backoff starting at 2s (honours `Retry-After` when present); supports `settings.semantic_scholar_api_key` for the keyed 1-RPS guarantee. Dice-0.6 title-similarity gate + year ±1 + surname-intersection guard block false positives (S2 happily matches "Climate of the Past, Present and Future" to "gravity-wave parametrization" on generic titles otherwise). **(5) `search_europepmc_by_title`** — 33M life-sci / Agricola / patents records, `TITLE:"..."` Lucene query, returns `{doi, pmid, pmcid, title, authors, journal}`. **(6) `search_arxiv_by_title`** — Atom-XML API `ti:"..." AND au:surname` search for preprints whose journal-version PDF doesn't print the arXiv ID. **(7) `search_datacite_by_title`** — relaxed `titles.title:(tok1 tok2 ...)` Lucene query (the strict phrase form returns 0 hits on real titles); returns DOIs for PANGAEA/Zenodo/NASA datasets. Cascade order in `_lookup`: XMP → fulltext-regex → title-recovery → Crossref-title → OpenAlex-title → S2 /match → Europe PMC → arXiv title → DataCite → arXiv-by-ID stub. Post-XMP/FT hit, a title-corroboration gate (Dice ≥ 0.5 against the DB title when the DB title isn't garbage) rejects inherited-XMP false positives (template PDFs, composite collections). `meta.source` now tracks both the DOI-discovery layer (`xmp_pdf` / `fulltext_regex` / `recovered_title`) and the resolver (`crossref` / `datacite` / `semantic_scholar_match`) via a `+`-joined string so the end-of-run per-source breakdown (`Per-source hits: crossref+recovered_title 18`) tells users which investment (key / email / quota) pays off on their corpus. New settings: `SEMANTIC_SCHOLAR_API_KEY` (optional), `UNPAYWALL_EMAIL` (optional, not yet wired — reserved for future DOI-page HTML validation). Documents row read switched to `LEFT JOIN documents d ON d.id = pm.document_id` so the PDF-read layers get `original_path`. **Measured on 217-paper no-DOI subset of the global-cooling corpus**: pre-change pipeline matched 2; post-change matched **45** (18 recovered_title, 13 xmp_pdf, 11 fulltext_regex, 2 crossref, 1 datacite) — a **22× improvement** in hit rate. The residual 172 no-matches are mostly foreign-language (Spanish volcanology), grey-literature blog posts, AIAA conference proceedings, and pre-DOI era papers — genuinely not recoverable from any DOI index. L1 suite green.

> **Phase 54.6.150 — Retrieval-density widener: wpc lerp from evidence pool size.** Ships RESEARCH.md §24 guideline 4 — the one flagged in the research brief as *"honest novel engineering, graded B as engineering / N/A as science"*. > **Phase 54.6.64 — replace `:name::type` with `CAST(:name AS type)` in five INSERTs.** User reported `sciknow book outline "Global Cooling"` crashed with *`ProgrammingError: syntax error at or near ":"`* at `:secs::jsonb`.
